================================================================================
  1. PROJECT OVERVIEW
================================================================================

Project Name   : Forensic Analysis Platform
Version        : 2.1.0
Purpose        : A full-stack web application for digital forensic disk-image
                 analysis. Users upload evidence files (RAW, DD, E01, IMG, ISO),
                 the backend extracts file-system artifacts, and a RAG-powered
                 AI chatbot lets investigators query the findings in natural
                 language. Professional PDF and DOCX reports are generated with
                 optional LLM-written executive summaries.

Key capabilities:
  - Upload and parse forensic disk images (.raw, .dd, .e01, .img, .iso, .bin, .dmg)
  - Recursive file-system scanning (NTFS, FAT32, ext4, etc.)
  - SHA-256 hash computation for every extracted file
  - Suspicious-file detection (executables, scripts, DLLs)
  - Deleted-file recovery metadata
  - Encryption detection (TrueCrypt, VeraCrypt, BitLocker, LUKS, PGP)
  - Network artifact extraction (hosts file, browser data, temp files)
  - Timeline generation (chronological modification events)
  - PDF and DOCX professional report export with LLM-generated narratives
  - RAG chatbot (ChromaDB + sentence-transformers + Llama 3 via Ollama)
  - Demo mode (no pytsk3/pyewf required) for testing and development

Frontend framework   : React 18 with TypeScript, bundled by Vite 5
Styling              : Tailwind CSS 3 + custom Halide design system (dark forensic theme)
Animations           : Framer Motion (navbar transitions), Canvas API (LetterGlitch bg)
Icons                : Lucide React
Backend framework    : Flask 2.3 with flask-cors for CORS
AI/LLM               : Llama 3 via Ollama (localhost:11434) — optional, falls back to rule-based
Embedding model      : sentence-transformers "all-MiniLM-L6-v2" (384-dim, runs locally)
Vector database      : ChromaDB (PersistentClient, cosine similarity)
Disk image parsing   : pytsk3 (The Sleuth Kit bindings) + pyewf (EnCase E01 format)
Report generation    : reportlab (PDF) + python-docx (DOCX)

Communication:
  - Frontend (Vite dev server :5173) proxies /api/* to Flask (:5000)
  - All API calls use the browser's native fetch() — no axios dependency
  - CORS configured for localhost:5173, localhost:3000, and 127.0.0.1 variants
  - File uploads sent as multipart/form-data via POST /api/upload
  - Analysis runs in a background Python thread; frontend polls GET /api/analysis/{id}/status
  - RAG chat uses POST /api/rag/chat with JSON body { query, top_k, use_llm }

================================================================================
  5. API ENDPOINT DOCUMENTATION
================================================================================

All endpoints are served by forensic_web_app.py at http://127.0.0.1:5000.
The Vite dev server proxies /api/* from :5173 to :5000.

--------------------------------------------------------------------------------

  GET /
  Purpose: Render main dashboard (Flask template fallback).
  Response: HTML page (templates/index.html)
  Called by: Direct browser access to :5000

--------------------------------------------------------------------------------

  GET /api/status
  Purpose: Get current analysis status (global state).
  Response: JSON {
    current: string|null,    // analysis ID
    progress: number,        // 0-100
    status: "idle"|"analyzing"|"completed"|"error",
    message: string,
    errors: string[],
    results: object|null,
    rag_collection: string|null
  }
  Called by: forensic-file-upload.tsx (health check),
             App.tsx (restore state on mount)

--------------------------------------------------------------------------------

  POST /api/upload
  Purpose: Upload a forensic disk image file.
  Input: multipart/form-data with field "file"
  Validation: Extension must be in {raw, dd, e01, img, iso, bin, dmg}
  Storage: Saved to uploads/{timestamp}_{filename}
  Response: JSON {
    success: true,
    message: "File uploaded successfully",
    filename: string,
    filepath: string
  }
  Error: 400 (no file / invalid type), 500 (server error)
  Called by: forensic-file-upload.tsx

--------------------------------------------------------------------------------

  POST /api/analyze
  Purpose: Start forensic analysis on an uploaded image.
  Input: JSON { image_path: string }
  Action: Spawns background thread running run_forensic_analysis()
  Response: JSON {
    success: true,
    analysis_id: string (timestamp format: YYYYMMDD_HHMMSS),
    message: "Analysis started"
  }
  Error: 400 (no path / file not found), 500 (server error)
  Called by: forensic-file-upload.tsx

--------------------------------------------------------------------------------

  GET /api/analysis/<analysis_id>/status
  Purpose: Poll status of a specific running analysis.
  Response: JSON {
    id: string,
    status: "idle"|"analyzing"|"completed"|"error",
    progress: number,
    message: string,
    errors: string[],
    results: object|null     // populated when status = "completed"
  }
  Error: 404 (analysis not found)
  Called by: forensic-file-upload.tsx (polling every 2 seconds),
             forensic-analysis-dashboard.tsx

--------------------------------------------------------------------------------

  GET /api/results
  Purpose: Get the latest completed analysis results (summary only).
  Response: JSON {
    timestamp: string,
    image_path: string,
    summary: { total_files, total_deleted_files, suspicious_files_count,
               encrypted_items_count, network_artifacts_count },
    extensions: { ".exe": 5, ".dll": 3, ... },
    encrypted_items: [...],
    network_artifacts: [...]
  }
  Error: 400 (no results available)
  Called by: forensic-analysis-dashboard.tsx

--------------------------------------------------------------------------------

  GET /api/export/<format>
  Purpose: Generate and download a PDF or DOCX report.
  Parameters: format = "pdf" | "doc" | "docx"
  Action:
    - PDF: Calls generate_pdf_report() → uses reportlab to build multi-page report

             forensic-report-viewer.tsx

================================================================================
  6. RAG PIPELINE EXPLANATION
================================================================================

The RAG (Retrieval-Augmented Generation) system is implemented entirely in
rag_engine.py and enables investigators to ask natural-language questions
about the forensic evidence.

--------------------------------------------------------------------------------
  6.1  DOCUMENT CHUNKING (chunk_forensic_report)
--------------------------------------------------------------------------------

The JSON analysis report is split into meaningful, self-contained text chunks:

  Chunk Type              Source Label          Content
  ──────────────────────  ────────────────────  ─────────────────────────────
  Overview                "overview"            Image path, timestamp, filesystem
                                                info, summary statistics
  Extension Statistics    "extension_stats"     File type distribution table
  Per-File Metadata       "file:{filename}"     Name, path, size, timestamps,
                                                inode, deleted status, hash,
                                                suspicious flag
  Encrypted Items         "encrypted_items"     Names, paths, types, sizes
  Network Artifacts       "network_artifacts"   Names, paths, types, sizes
  Timeline Batches        "timeline_batch_{n}"  Groups of 5 timeline events

Each chunk is classified into an artifact_type using rule-based pattern matching:
  - browser_history, usb_logs, login_events, installed_programs,
    network_connections, filesystem, general

--------------------------------------------------------------------------------
  6.2  EMBEDDING GENERATION (vectorize_and_store)
--------------------------------------------------------------------------------

  Model: sentence-transformers "all-MiniLM-L6-v2"
  - 384-dimensional vectors
  - ~22 MB download on first run
  - Runs entirely locally (no API key needed)
  - Lazy-loaded to avoid startup delay

  Process:
  1. All chunk texts are batch-encoded into 384-dim float vectors
  2. Deterministic IDs generated via SHA-256 of chunk text
  3. Metadata attached: source, analysis_id, artifact_type, source_file, page_id

--------------------------------------------------------------------------------
  6.3  VECTOR DATABASE STORAGE (ChromaDB)
--------------------------------------------------------------------------------

  Database: ChromaDB PersistentClient
  Storage path: ./chromadb_store/
  Collection name: "forensic_{analysis_id}" (e.g., "forensic_20260304_094921")
  Distance metric: cosine similarity (hnsw:space = "cosine")
  
  Each document is stored with:
  - id: deterministic hash-based ID
  - document: original chunk text
  - embedding: 384-dim float vector
  - metadata: { source, analysis_id, artifact_type, source_file, page_id }

  Old collections with the same name are deleted before re-indexing.

--------------------------------------------------------------------------------
  6.4  RETRIEVAL (retrieve_context) — 5-step pipeline
--------------------------------------------------------------------------------

  Step A: Semantic Search (always)
    - Encode user query into 384-dim vector
    - Query ChromaDB for top-K nearest neighbors by cosine distance
    - Filter by DISTANCE_THRESHOLD (0.75 max cosine distance)

  Step B: Query Intent Detection
    - Rule-based classifier (_detect_query_artifact) checks query keywords
    - Maps to artifact category: browser_history, usb_logs, login_events,
      installed_programs, network_connections, filesystem
    - Returns None if no specific category detected

  Step C: Hierarchical Filtered Search (if category detected)
    - Queries ChromaDB with metadata filter: { artifact_type: detected_category }
    - Returns chunks matching both semantic similarity AND artifact type

  Step D: Hybrid Merge & Deduplication
    - Round-robin interleaves filtered hits (priority) and semantic hits
    - Deduplicates by first 200 chars of text
    - Sorts by distance (best first), trims to top_k

  Step E: Context Compression
    - Splits retrieved chunks into individual sentences
    - Embeds all sentences and computes cosine similarity with query
    - Keeps only sentences above similarity_threshold (0.25)
    - Returns top 8 most relevant sentences, grouped by source
    - Skips compression for small result sets (< 500 chars)

--------------------------------------------------------------------------------
  6.5  LLM RESPONSE GENERATION (ask)
--------------------------------------------------------------------------------

  Step 0: Relevance Guard
    - Rejects off-topic queries (greetings, jokes, general knowledge)
    - Uses _IRRELEVANT_PATTERNS (exact match) and _FORENSIC_KEYWORDS (keyword check)
    - Returns canned rejection message for irrelevant queries

  Step 1: Prompt Construction (build_llm_prompt)
    - Uses Llama 3 instruct format: <|begin_of_text|> / system / user / assistant
    - System prompt enforces strict forensic-only responses
    - User section contains: forensic report context + user question

  Step 2: LLM Call (_call_ollama)
    - POST to http://localhost:11434/api/generate
    - Model: "llama3", temperature: 0.3, max tokens: 300
    - Timeout: 60 seconds

  Step 3: Fallback (generate_fallback_answer)
    - If Ollama is unavailable, uses rule-based pattern matching
    - Intent handlers for: file count, suspicious files, deleted files,
      encrypted items, timeline, network, summary/overview
    - Generic fallback: returns top-3 most relevant chunks

  Response includes:
    - answer: Generated text
    - context: Source chunks used (text + source label)
    - collection: ChromaDB collection name
    - rejected: Whether query was off-topic

--------------------------------------------------------------------------------
  6.6  COMPLETE RAG FLOW DIAGRAM
--------------------------------------------------------------------------------

  User types question in forensic-rag-chat.tsx
    │
    ▼
  POST /api/rag/chat  { query: "What suspicious files were found?" }
    │
    ▼
  _is_relevant_query() → PASS (contains "suspicious" keyword)
    │
    ▼
  _get_embed_model().encode(query) → [0.234, -0.156, 0.891, ...] (384-dim)
    │
    ├─────────────────────────────────────────────┐
    ▼                                             ▼
  Semantic search                         _detect_query_artifact()
  ChromaDB.query(embedding, n=5)          → "filesystem" category
    │                                             │
    │                                             ▼
    │                                    Filtered search
    │                                    ChromaDB.query(embedding, n=5,
    │                                      where={artifact_type: "filesystem"})
    │                                             │
    └──────────────┬──────────────────────────────┘
                   ▼
  _merge_and_deduplicate() → Round-robin interleave → deduplicate → sort
                   │
                   ▼
  _compress_context() → Split to sentences → embed → cosine similarity
                       → Keep top 8 above threshold 0.25
                   │
                   ▼
  build_llm_prompt() → Llama 3 instruct format with context + question
                   │
                   ▼
  _call_ollama() → POST localhost:11434/api/generate
                   │
                   ├── Success → LLM answer
                   └── Failure → generate_fallback_answer() (rule-based)
                   │
                   ▼
  Response JSON { answer, context, collection, rejected: false }
                   │
                   ▼
  forensic-rag-chat.tsx renders answer with source citations


================================================================================
  7. END-TO-END WORKFLOW
================================================================================

STEP 1: USER OPENS APPLICATION
  - Navigates to http://localhost:5173 (Vite dev server)
  - Vite serves the React SPA
  - Landing page renders: forensic-analysis-hero.tsx (3D parallax)
  - User clicks "BEGIN ANALYSIS" → navigates to /analysis

STEP 2: ANALYSIS PAGE LOADS
  - App.tsx lazy-loads: ForensicFileUpload, ForensicAnalysisDashboard,
    ForensicRagChat, LetterGlitch (canvas bg), NavBar
  - NavBar renders with tabs: Home, Upload, Analysis, Report
  - Default tab: Upload
  - App.tsx fetches GET /api/status to check for prior completed analysis

STEP 3: USER UPLOADS EVIDENCE FILE
  - User selects disk image file in ForensicFileUpload
  - Client-side validation: checks extension (.raw, .dd, .e01, etc.)
  - POST /api/upload sends file as multipart/form-data
  - Flask saves file to uploads/{timestamp}_{filename}
  - Flask returns { success, filename, filepath }

STEP 4: ANALYSIS IS TRIGGERED
  - Frontend sends POST /api/analyze with { image_path: filepath }
  - Flask spawns background thread: run_forensic_analysis(path, analysis_id)
  - Returns immediately: { success, analysis_id, message }

STEP 5: BACKGROUND ANALYSIS PIPELINE (run_forensic_analysis)
  Progress 5%   — Initialize AdvancedForensicAnalyzer
                  Auto-detect demo mode if pytsk3/pyewf unavailable
                  If demo + real file exists: inject real file metadata + SHA-256
  Progress 25%  — open_disk_image() — mount RAW/E01 via pytsk3/pyewf (or simulate)
  Progress 35%  — detect_partitions() — find allocated partitions
  Progress 45%  — open_filesystem() — mount filesystem (NTFS/FAT/ext4)
  Progress 55%  — recursively_scan_files() — walk entire filesystem tree
                  Extract: name, path, size, inode, timestamps, deleted flag
                  Count file extensions, build timeline events
  Progress 65%  — compute_file_hashes() — SHA-256 for every file
  Progress 75%  — detect_encryption() — scan for encrypted volumes
                  detect_network_artifacts() — find network-related files
  Progress 85%  — Generate reports:
                  → {id}_report.json     (complete structured data)
                  → {id}_timeline.txt    (chronological events)
                  → {id}_extensions.txt  (file type distribution)
                  → {id}_summary.txt     (human-readable summary + optional LLM)
  Progress 95%  — RAG vectorization:
                  → chunk_forensic_report() splits JSON into chunks
                  → sentence-transformers encodes chunks to 384-dim vectors
                  → ChromaDB stores vectors in chromadb_store/
  Progress 100% — Analysis complete, results stored in global analysis_status

STEP 6: FRONTEND POLLS FOR COMPLETION
  - ForensicFileUpload polls GET /api/analysis/{id}/status every 2 seconds
  - Updates progress bar (0% → 100%)
  - On "completed": calls onAnalysisComplete(result)
  - App.tsx stores result in analysisResult state

STEP 7: RESULTS DISPLAYED
  - ForensicAnalysisDashboard renders stats grid:
    Total files | Deleted files | Suspicious | Encrypted | Network | File types
  - ForensicReportViewer shows tabs: Summary, Timeline, Files
  - ReportDownloadButtons appear: Download PDF | Download DOCX

STEP 8: USER DOWNLOADS REPORT (optional)
  - Clicks "Download PDF" → GET /api/export/pdf
  - Flask calls generate_pdf_report():
    → Sends data summary to Ollama for LLM executive summary
    → Sends data summary to Ollama for LLM detailed findings
    → Builds multi-page PDF with reportlab:
       Cover page, Evidence Info, Executive Summary, Statistics,
       Extension Distribution, Suspicious Files (red table),
       Deleted Files, Encrypted Items, Network Artifacts,
       Detailed Findings, Disclaimer
  - Returns binary PDF with Content-Disposition: attachment
  - Same flow for DOCX via generate_doc_report() with python-docx

STEP 9: USER INTERACTS WITH RAG CHATBOT
  - User switches to "Analysis" tab
  - ForensicRagChat polls GET /api/rag/status until { ready: true }
  - User types question or clicks quick-prompt chip
  - POST /api/rag/chat with { query, top_k: 5, use_llm: true }
  - Backend: relevance check → retrieve context → LLM/fallback → return answer
  - Frontend renders answer with markdown formatting and source citations
  - User can ask follow-up questions (chat history maintained client-side)

STEP 10: NEW ANALYSIS (optional)
  - User can click reset or upload a new file
  - GET /api/reset clears global state
  - Previous ChromaDB collection persists until overwritten
  - New upload starts the cycle again from Step 3
